# CrossFrame Review Report Template

本模板用于用户显式要求完整评审报告、只评审已有输出，或质量闸发现硬失败而必须说明阻断原因的场景。若本次只是 `crossframe-suite -> crossframe-essay -> crossframe-review` 默认成文链路的质量闸，且文章层未关闭，不得用本模板替代最终的 `结构洞察底稿` 和 `文章正文`。

```markdown
## 复核清单

- 是否明确对象、事实边界和尺度窗口：
- 是否有机制候选和反向条件：
- 是否保留判断档位：
- 是否触发并保留 v5.0 源结构连续联读包：
- 是否生成并复用 `v5-read-state-capsule`：
- 中心命题、机制候选、高风险概念和行动边界是否能回指胶囊源锚点：
- 是否生成 `claim ledger`：
- 正文中心命题、机制句、高风险概念、行动建议、概率排序和点睛句是否都有对应 `claim_id`：
- `claim_id` 是否能回指事实、源锚点、概念契约、来源台账、判断档位和撤回条件：
- v5 DLC 是否触发；是否登记 `score_visibility`：
- 分值是否只用于降档、补证、阻断发布或行动上限，而没有被用于处置、排名、公开定性、发布通过或 `substantive_pass`：
- 普通关系、组织或公共议题是否在用户未要求审计/量化时自动外显分数：
- 是否出现生成层质量自我通过、完全通过、A档、合格或 `substantive_pass`：
- 开放断言、可撤回判断、最高开放断言、不能终局裁决、当前只能说、不能证明什么或撤回条件是否触发 `contract-open_assertion`：
- 前台表是完整台账还是命题台账摘要；是否缺少 source_anchor、mechanism_id、concept_contract、source_ledger_id 或 publish_boundary：
- mechanism_candidate 是否有 mechanism_id：
- 标题、首段、结尾和点睛句是否强于对应 `claim_id`：
- 技法读取是否只写本次所选技法，而不是“所有技法已读取”：
- 模式/角色选择器和文章类型选择器是否完整展示且等待回复：
- 来源台账是否区分来源用途、证据档位和仍不能证明什么：
- 来源台账是否十字段完整、时间可复核、使用位置可定位：
- 写作技法是否只改变段落动作，不新增事实、强判断或框架原义：
- 每个技法是否有正文短摘/段落编号：
- 是否避免人格审判和尺度洗白：
- 是否处理引用、查源和 AI 材料的证据成本：
- 文章是否先有结构洞察底稿：
```

## 极简输出

当用户要求极简时：

```markdown
结论：不合格，等级 C。主要问题是【问题】，证据在【定位】；先补【最关键修复动作】，再复核判断档位。
```

## 语气

评审要直截了当，但不羞辱作者。批评对象是输出质量、证据边界和推理链，不是作者人格。
