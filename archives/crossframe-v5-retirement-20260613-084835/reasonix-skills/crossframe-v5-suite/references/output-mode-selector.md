# 输出模式与角色选择器

本文件定义 CrossFrame v5 分析的输出模式（推断风险偏好）和角色（目标受众与分析立场），作为正交于推理链的第二个维度。模式×角色的组合只改变前台表达风格和推断边界，不降低概念保真、机制候选、判断档位和连续联读的任何要求。

## 一、输出模式（4档）—— 决定"冒多大推断风险"

| 模式 | 触发词 | 内部行为 |
|------|-------|---------|
| 🔵 **保守** | "保守一点""只确认最稳的""稳一点" | 强制留两处边界/反例；开放断言以上不输出；所有判断写撤回条件；机制候选至少两个且每个必须有不成立的反向信号 |
| 🟢 **客观** | "客观""不站队""平衡""正反都看"（默认） | 先平衡呈现正反与竞争机制；若证据明显支持某个方向，可输出“客观开放断言”：给出当前条件性判断、支持证据、替代解释、反向条件、撤回条件和行动边界；不得把“客观”理解为强制 50/50 或永远不判断 |
| 🟠 **激进** | "激进""最大胆的判断""直说" | 允许单一主机制（但必须注明"此为当前最稳假说，竞争机制见底稿"）；允许推断链延伸两层；边界段缩短为一句话（但仍必须存在）；撤回条件仍需写清 |
| 🔴 **批判** | "批判""找问题""质疑""反驳""有什么问题" | 以反例、弱信号、隐藏代价为主要组织方式；肯定性内容压缩至总体上不超过全文 20%；强制包含"最强反方"段（即使作者本人不同意）；指出盲点、利益偏向或副作用 |

### 客观开放断言

`客观` 模式下的平衡，是对证据、机制候选和撤回条件的平衡，不是把所有结论压成均势。当题目要求“接受/拒绝”“支持/反对”“该不该做”等方向性回答时：

- 可以给出当前最稳的条件性判断，但必须同时写出至少一个有实质竞争力的替代解释。
- 必须说明判断依赖的题设、证据和价值前提；撤回条件不能只写成形式化保留。
- 如果通过拆题、重定义问题、增加“改良版选项”或调整题设来回答，必须显式声明：原题设下的回答是什么，改写后的回答是什么。
- 方向性判断一旦可能影响权利、名誉、资源、处置、求助路径或现实行动，触发 `v5-judgment-responsibility-pack`。

## 二、角色选择（6种）—— 决定"为谁分析、用什么语言"

| 角色 | 触发词 | 后台推理引擎 | 前台表达风格 |
|------|-------|-------------|-------------|
| 🎓 **1. 学术专家** | "学术""严谨""引用理论""学者视角" | 诊断协议 + 概念解释协议 + 概念上升协议（必开） | 概念定义先于判断；保留条件与边界；可引用经典但只能意译/思想映射；结论写"当前证据支持……但若出现X则需修正" |
| 🔧 **2. 实践工匠** | "实操""怎么做""经验""落地" | 诊断协议 + 低条件行动协议 | 步骤/工具/周期/成本/常见坑点；用一线案例说明成败关键；结论写"先做A，观察B，如果C则停/改" |
| 🎯 **3. 战略决策者** | "战略""决策""优先级""投入产出" | 诊断协议 + 递进协议 + 治理连续性协议 | 目标-资源-风险-收益平衡；"要不要做、投入多少、何时止损"；全局优先级排序；保留退出条件和替代路径 |
| 📣 **4. 大众传播** | "通俗""易懂""给普通人看""简单说" | 诊断协议 + 表达翻译协议 | 比喻/故事/直观感受；避免术语（必须用的只做括号注释）；给可直接记住的建议；情绪共鸣但不煽动 |
| 🔍 **5. 批判反思者** | "批判反思""质疑前提""反方""盲点" | 诊断协议 + 命题验证协议 + 反俘获协议 | 质疑前提、寻找反例和潜在代价；指出盲点、利益偏向或副作用；对冲群体共识；强制包含"如果这个判断错了会怎样"段 |
| 🔮 **6. 未来探索者** | "未来""趋势""5年后""推演""演变" | v5 连读包 + 推演协议（`inference-protocol.md`）+ 生命周期协议（`lifecycle-diagnosis-protocol.md`）+ 表达翻译协议 | 基于状态坐标、反馈写回、承接者生成和边缘信号做条件路径图；多分支展开（至少两个分歧路径）；每个路径写触发条件、反向信号、保护变量和撤回条件 |

### 未来探索者角色的内部路由（自动触发）

当角色 = 6（未来探索者）时，自动触发以下路由：

1. 先触发 `v5-use-boundary-low-power-pack`、`v5-seven-gates-diagnosis-pack`、`v5-root-evolution-deep-pack`；按对象追加证据、观察反身性、公共权力、判断责任或行动转移连读包。
2. 读取 `../crossframe-v5/protocols/inference-protocol.md`（推演协议——条件路径、状态坐标、触发条件、反向信号、撤回条件）。
3. 读取 `../crossframe-v5/protocols/lifecycle-diagnosis-protocol.md`（生命周期阶段判断——作为推演坐标，不作为命运结论）。
4. 读取 `../crossframe-v5/protocols/expression-translation-protocol.md`（前台表达翻译——把后台推演链翻译成人话）。
5. 输出模板使用 `../crossframe-v5/templates/inference-output.md`（从底稿转译为可审计的条件路径图）。
6. 至少展开两个分歧路径，每个路径写清："触发条件是什么、阶段坐标如何移动、谁在承接成本、保护变量是什么、什么信号会让这个路径关闭"。
7. 只有证据、尺度和时间窗口足够时才展开 5-10 年；否则使用更近窗口，并把 5-10 年部分标记为高度假设。

### 各角色的默认声口

| 角色 | 默认声口 | 说明 |
|------|---------|------|
| 学术专家 | 中性分析体 | 不启用现代编辑底色；结论保留条件和边界 |
| 实践工匠 | 中性分析体（可带决定语气） | "建议先做X，观察Y" |
| 战略决策者 | 中性分析体（可带决定语气） | "优先级排序：A>B>C" |
| 大众传播 | 可启用编辑底色（答复体/评论体） | 通俗、亲切、有共鸣 |
| 批判反思者 | 中性分析体 | 质疑和反例本身构成文章结构 |
| 未来探索者 | 可启用编辑底色（评论体） | 叙事化推演，但保留分支条件 |

用户显式要求"亲切/编辑口吻/答复体"时，覆盖上述默认，对所有角色启用现代编辑底色。

### 中性分析体的边界

中性分析体不是冷淡体。它允许清楚、平实、有正常人的温度，也允许为解释机制而使用简短比喻；但温度不能替代证据，不能改变判断档位，不能把读者情绪推成结论。

在未启用编辑底色的角色（学术专家/批判反思者）中，如果启用明显共情语调、编辑底色、直接劝慰段、读者求助提醒或“改良版按钮”这类创造性转向，必须在底稿或正文中声明原因、边界和题设改写位置。普通的礼貌、清晰和不冷漠不算越界。

## 三、主题敏感度（topic_sensitivity）—— 独立于模式和角色

模式决定推断风险，角色决定表达对象；`topic_sensitivity` 决定承接和保护优先级。它不改变事实判断，但改变入口顺序、底稿格式和质量闸。

| 敏感度 | 典型主题 | 强制要求 |
| --- | --- | --- |
| `low` | 抽象概念、一般随笔、低风险思想实验 | 可直接进入机制分析；仍需写题设和撤回条件 |
| `normal` | 普通关系、团队、组织、制度分析 | 正常执行对象、事实、尺度、机制候选和责任链 |
| `vulnerable` | 痛苦、绝望、创伤、无法退出、亲密伤害、求助暗示、主体安全 | 第一段先接住人；不得把问题当纯逻辑游戏；优先保护痛苦、安全、最小自主和低风险下一步 |
| `high-stakes` | 法律、医疗、金融、处分、名誉、资源分配、公共记忆、真实机构强判断 | 默认审计型底稿；必须完成证据边界、判断责任、查源/反证/申诉入口；不得用文章温度覆盖高责任保护 |

`vulnerable` 和 `high-stakes` 可以叠加。叠加时，先承接主体处境，再进入审计型判断。

## 四、模式×角色 的正交关系

深度（完整链路）和模式+角色是两个独立轴：
- **完整链路**：所有任务走同一完整链路 `crossframe-v5 → [sibling skills] → essay → review → final assembly`
- **模式**：决定推断风险偏好（保守↔批判）
- **角色**：决定目标受众和分析立场（学术↔未来）

任何角色都可以和任何模式组合。例如：
- "保守 + 学术专家"：最稳的学术判断，多重边界
- "激进 + 未来探索者"：大胆推演 5-10 年演变
- "批判 + 批判反思者"：以反例、盲点和隐藏代价为组织方式的思想文章
- "客观 + 大众传播"：平衡的通俗解释

## 五、默认值

未指定时：
- 模式 = 🟢 客观
- 角色 = 🎓 学术专家
- 主题敏感度 = 根据题材自动判定；不能因默认学术专家而忽略 `vulnerable` 或 `high-stakes`

用户用"保守/客观/激进/批判"关键词切换模式，用"学术/实操/战略/通俗/批判反思/未来"或数字 1-6 切换角色。

## 六、与现有声口系统的关系

本选择器是 suite 调度协议的正交选择层。声口不由文章层单独决定，而由模式、角色、主题敏感度和必要连读包共同约束。详见 `crossframe-v5-suite/protocols/suite-dispatch-protocol.md` 的入口选择与调度规则。
