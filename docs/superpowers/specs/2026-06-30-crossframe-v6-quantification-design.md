# CrossFrame v6.0 半量化结构判断系统设计

## 目标

将《跨尺度结构诊断框架 v5.0》迭代为 v6.0：在不把复杂现实压成总分或精确预测的前提下，建立一套可审计的半量化结构判断系统。

v6.0 的量化层只承担四类责任：

- 校准判断：说明一个结构变量在当前材料中处于什么程度，而不是给对象定性。
- 暴露不确定性：把证据缺口、反例压力、复核条件和撤回条件显式化。
- 限制行动上限：证据和七闸不足时自动降级，不允许分数越权授权行动。
- 触发复核和写回：发现反例、评分分歧或误用时，必须改变台账、案例库、工具或文本边界。

## 非目标

v6.0 不做以下事情：

- 不输出总分、人格分、组织健康总分、关系质量总分或文明阶段总分。
- 不输出精确预测概率，例如“73% 会崩溃”。
- 不把评分表、AI 输出、内部压力测试分数、验证脚本通过或案例库覆盖率当作现实安全证明。
- 不让量化表单单独用于开除、封禁、惩戒、拘束、拒绝服务、剥夺资源、资格判断或人格定性。
- 不把自然科学、系统论、心理测量或风险评估术语反向改写 CrossFrame 中文概念。

## 设计原则

1. 数字服从台账

   所有分值、等级、隶属度和更新方向必须回指 `source_id`、`claim_id`、`mechanism_id`、概念契约和撤回条件。不能回指的数字只能作为草稿线索。

2. 评分只能降级，不能自动升级

   高分不能绕过七闸、强判断八件套、低权力保护、来源台账、概念契约和外部复核。任一关键闸口 `fail` 或 `blocked` 时，最高判断档位必须下降。

3. 结构剖面优先于总分

   输出应呈现多维剖面，例如“反馈写回阻断高、责任链可追踪低、行动上限清楚、反例入口不足”，而不是将这些维度加权成一个总分。

4. 校准锚点先于评分

   每个可评分维度必须先定义 0-4 的现实锚点、可观察信号、证据要求和误用风险。没有锚点的维度不得进入正式工具。

5. 案例库用于失败发现，不用于自我证明

   案例库验证只能说明某套规则在既有案例中如何表现、哪里误判、哪里需要降级；不能证明框架现实可靠，也不能证明框架比替代方法优越。

6. 评分者一致性验证协议稳定性，不验证现实真相

   评分者一致性只说明不同使用者是否按同一规则理解材料。它不证明结论真实，也不能替代受影响位置反馈。

## 总体架构

v6.0 半量化系统分为七层。

| 层级 | 产物 | 作用 |
| --- | --- | --- |
| 构念图谱 | `construct-map.md` | 定义可观察结构变量、相邻概念和禁止合并项 |
| 七闸半量化 | `seven-gates-quant-rubric.md` | 将对象、证据、尺度、责任、观测、权力、行动七闸转成状态码加锚点评分 |
| 证据台账扩展 | `evidence-ledger-v6.md` | 增加证据直接性、独立性、可复核性、反例压力和诊断力字段 |
| 校准锚点 | `calibration-anchors.md` | 为每个维度定义 0-4 锚点和撤回条件 |
| 机制候选更新 | `mechanism-update-rules.md` | 记录证据如何增强、削弱、保留、待证或撤回机制候选 |
| 判断档位与行动上限 | `judgment-action-matrix.md` | 将七闸、证据和机制状态映射为最高允许判断档位和行动上限 |
| 反例与治理写回 | `falsification-governance.md` | 定义反例、评分分歧、误用案例如何写回文本、工具和版本日志 |

这些文件后续可以进入 `skills/crossframe/references/`、`skills/crossframe/worksheets/`、`skills/crossframe/schemas/` 和 `docs/`。第一阶段先在设计文档中冻结结构，不直接改可安装 skill。

## 构念图谱

构念图谱定义哪些结构变量可以进入 v6.0 半量化层。第一批构念应从 v5.0 已有诊断动作中抽取，不新增本体概念。

优先构念：

| 构念 | 来源接口 | 现实问题 | 不可误用为 |
| --- | --- | --- | --- |
| 对象边界清晰度 | intake / 对象闸 | 本次诊断对象、时间窗和不判断事项是否清楚 | 对象价值判断 |
| 证据可支撑度 | evidence ledger / 证据闸 | 材料能支撑什么判断档位 | 来源权威崇拜 |
| 尺度一致性 | 尺度闸 | 是否发生跨尺度偷换 | 宏大叙事正确性 |
| 责任链可追踪度 | mechanism candidates / 责任闸 | 谁造成、受益、承接、能改变 | 人格责任总分 |
| 观测反身性风险 | 观测闸 | 诊断、评分、发布是否改变对象 | 对象本质判断 |
| 低权力反例入口 | 权力闸 | 受影响者是否能安全申诉、补证、反驳 | 形式程序存在证明 |
| 行动上限清晰度 | action ceiling / 行动闸 | 当前最多能做什么、不能做什么 | 行动授权 |
| 反馈写回程度 | 核心概念 / 回流 | 反馈是否改变规则、资源、角色、边界或记忆 | 满意度 |
| 修复窗口可行度 | healing / low condition action | 是否存在低风险、可撤回、可观察动作 | 和解命令 |
| 反例写回强度 | framework governance | 反例是否改变文本、流程、工具或边界 | 反例收集数量 |

构念升格条件：

- 能改变对象界定、证据判断、机制候选、责任链、行动边界或输出表达。
- 能写出可观察信号。
- 能写出最低证据要求。
- 能写出最容易被怎样误用。
- 能映射到现有 v5.0 概念卡、工作表、台账或连读包。

## 七闸半量化模型

七闸继续保留 `pass / weak / fail / blocked`，新增 0-4 锚点评分。状态码决定门禁，分值只解释状态来源。

通用评分锚点：

| 分值 | 含义 | 最高用途 |
| --- | --- | --- |
| 0 | 缺失、不可判断或材料被阻断 | 只能记录缺口 |
| 1 | 有低成本线索，但边界、来源或反例不足 | 轻量观察 |
| 2 | 可形成开放断言，但必须保留替代解释和撤回条件 | 开放断言 |
| 3 | 可支撑完整诊断，但高责任场景仍需追加强判断八件套 | 完整诊断 |
| 4 | 材料、反例、复核和行动边界较完整 | 可申请强判断流程，不自动通过 |

门禁规则：

- 任一闸 `blocked`：不得发布强判断或行动建议。
- 任一闸 `fail`：最高轻量观察、补证清单或低条件行动。
- 任一闸 `weak`：最高开放断言，除非补证后重跑。
- 证据闸低于 3：不得强判断。
- 权力闸低于 3：不得公开施压、处置或要求低权力主体继续暴露。
- 行动闸低于 3：不得给出处置、退出、追责、公开建议。
- 高责任主题即使七闸均为 4，也必须进入强判断八件套和外部复核条件检查。

## 七闸评分草案

### 对象闸：对象边界清晰度

| 分值 | 锚点 |
| --- | --- |
| 0 | 无法说明诊断对象、时间窗口和不判断事项 |
| 1 | 只有泛泛对象，例如“这个系统”“这段关系”，缺少边界 |
| 2 | 对象和时间窗口基本清楚，但不判断事项或受影响对象不足 |
| 3 | 对象、边界、时间窗口、不判断事项和受影响对象清楚 |
| 4 | 以上均清楚，并能说明对象边界变化时如何重跑诊断 |

### 证据闸：证据可支撑度

| 分值 | 锚点 |
| --- | --- |
| 0 | 来源、证据、解释和判断混在一起 |
| 1 | 有单方叙述、低成本声明或 AI/过程性产物 |
| 2 | 有可追溯来源和部分反例搜索，可支撑开放断言 |
| 3 | 有多源材料、中高成本证据、反例检查和缺口说明 |
| 4 | 有高成本证据、外部复核、来源台账、反例材料和撤回条件 |

### 尺度闸：尺度一致性

| 分值 | 锚点 |
| --- | --- |
| 0 | 不说明尺度窗口 |
| 1 | 尺度在个人、关系、组织、制度之间滑动 |
| 2 | 尺度基本可见，但跨尺度转译证据不足 |
| 3 | 尺度窗口、转移条件和不可外推边界清楚 |
| 4 | 多尺度链条清楚，且低尺度痛苦和责任没有被高尺度叙事抹掉 |

### 责任闸：责任链可追踪度

| 分值 | 锚点 |
| --- | --- |
| 0 | 无法识别谁造成、受益、承接或能改变 |
| 1 | 只描述氛围、态度或价值冲突 |
| 2 | 能识别部分承接者和成本，但改变条件不清 |
| 3 | 造成、受益、承接、改变条件和责任边界基本清楚 |
| 4 | 责任链、授权链、反馈链和干预边界均可追踪 |

### 观测闸：观测反身性风险

| 分值 | 锚点 |
| --- | --- |
| 0 | 未考虑诊断、评分、发布会改变对象 |
| 1 | 仅形式上提醒反身性 |
| 2 | 能列出表演、污名、报复或自证风险 |
| 3 | 已区分诊断前基线、观测中反应、发布后反应 |
| 4 | 已设定分层发布、延迟发布、复核入口和停止条件 |

### 权力闸：低权力反例入口

| 分值 | 锚点 |
| --- | --- |
| 0 | 低权力主体无法安全表达或提交反例 |
| 1 | 只有名义申诉入口，无反报复保护 |
| 2 | 有部分入口，但独立性、匿名性或反馈写回不足 |
| 3 | 有安全申诉、代理提交、反报复和补证路径 |
| 4 | 低权力反例能改变判断、行动上限、工具和版本写回 |

### 行动闸：行动上限清晰度

| 分值 | 锚点 |
| --- | --- |
| 0 | 判断没有行动边界 |
| 1 | 只写“建议关注”，没有停止条件 |
| 2 | 能指导低风险观察、记录或补证 |
| 3 | 行动上限、撤回条件、停止条件和扩大条件清楚 |
| 4 | 行动边界已与证据档位、权力风险和复核义务绑定 |

## 证据台账扩展

v6.0 应在现有 `source ledger` 和 `evidence ledger` 基础上增加半量化字段。

新增字段：

| 字段 | 取值 | 用途 |
| --- | --- | --- |
| evidence_cost | low / medium / high | 证据成本 |
| directness | direct / indirect / background | 是否直接支持 claim |
| independence | independent / related / same_source_family | 来源独立性 |
| verifiability | verifiable / partly_verifiable / not_verifiable | 可复核性 |
| counterevidence_status | searched / partial / missing / unsafe_to_collect | 反例搜索状态 |
| diagnostic_force | weak / moderate / strong / decisive_candidate | 对机制候选的诊断力 |
| mechanism_edge | mechanism_id + edge_id | 支持机制链的哪一段 |
| downgrade_reason | text | 为什么不能升级 |
| withdrawal_trigger | text | 什么事实触发撤回 |

使用规则：

- 低成本证据不得单独支撑强判断。
- 同源材料不能简单相加。
- AI/过程性产物默认低成本，除非它只是整理原始证据且原始证据已进入台账。
- 热度、沉默、投诉减少、满意度、内部评分只能作为线索，不得作为现实修复证明。
- 没有“不能证明什么”的来源，不得进入公共判断或强判断。

## 校准锚点规则

每个维度的正式评分必须有校准卡。

校准卡字段：

```text
construct_id:
中文名:
定义:
不是什么:
相邻概念:
0锚点:
1锚点:
2锚点:
3锚点:
4锚点:
最低证据:
高责任升级条件:
常见误用:
撤回条件:
对应 v5 源锚点:
对应工作表:
对应 schema 字段:
```

校准纪律：

- 不用样本均值、使用者直觉或便利数据直接切分锚点。
- 先定概念定义和现实锚点，再评分。
- 可用模糊集隶属度表达“高度属于某结构状态”，但不得转写为对象总分。
- 每次校准规则变更必须写版本日志。

## 机制候选更新

机制候选不做总排序，不输出“某机制 82 分”。v6.0 使用局部更新。

更新方向：

| 更新 | 含义 |
| --- | --- |
| strengthen | 证据增强该机制边 |
| weaken | 证据削弱该机制边 |
| neutral | 证据相关但诊断力不足 |
| pending | 需要补证 |
| withdraw | 关键反例成立，应撤回该机制候选或机制边 |

机制候选字段扩展：

```text
mechanism_id:
mechanism_edge_id:
机制描述:
支持事实:
反向证据:
证据诊断力:
更新方向:
当前状态:
能解释什么:
不能解释什么:
可观察预测:
撤回条件:
对应 claim_id:
```

规则：

- 一条证据只能更新它实际命中的机制边。
- smoking-gun 式强证据只能增强命中的边，不能替整套结构背书。
- hoop-test 式必要证据失败时，应削弱或撤回对应机制候选。
- 没有反向证据和“不能解释什么”的机制候选，最高开放断言。

## 判断档位与行动上限映射

v6.0 的判断档位仍沿用现有 normalized values：

- `light_observation`
- `open_assertion`
- `full_diagnosis`
- `strong_judgment`
- `low_condition_action`
- `exit_transfer`

映射规则：

| 条件 | 最高档位 | 行动上限 |
| --- | --- | --- |
| 对象不清或证据混乱 | light_observation | 澄清问题、整理材料 |
| 证据不足但边界清楚 | open_assertion | 观察、记录、补证 |
| 有中高成本证据和机制候选 | full_diagnosis | 低风险调整、复核建议 |
| 高责任且八件套完整 | strong_judgment candidate | 仍需外部复核，不自动处置 |
| 问题紧急但证据不完整 | low_condition_action | 可撤回、低风险、可观察动作 |
| 权力封闭或内部修复不可行 | exit_transfer | 保护、证据保存、外部承接 |

强制降档：

- 任一七闸 `fail` 或 `blocked`。
- 证据闸低于 3。
- 权力闸低于 3 且涉及低权力主体。
- 行动闸低于 3。
- `claim ledger` 缺失。
- 高风险概念缺少概念契约。
- 来源台账缺少“不能证明什么”。
- 反例搜索缺失且判断会影响资格、名誉、资源、权利或公共记忆。

## 反例、撤回和版本写回

反例不是附录材料，而是治理事件。

反例分级：

| 反例类型 | 处理 |
| --- | --- |
| 轻微反例 | 加入边界说明 |
| 重复反例 | 判断降级或校准锚点修订 |
| 高伤害反例 | 暂停相关用法并启动误用复核 |
| 系统性反例 | 引入替代机制或替代框架竞争 |
| 误用反例 | 发布阻断、工具红线或模板修改 |
| 不可修复案例 | 退出转移、保存演化记忆 |

每条反例至少登记：

```text
counterexample_id:
关联 claim_id / construct_id / mechanism_id:
材料来源:
反例类型:
影响范围:
文本后果:
工具后果:
行动上限变化:
版本写回:
```

## 案例库验证

案例库用于发现失败模式和校准问题，不用于证明框架正确。

案例分类：

- 成功案例
- 失败案例
- 早期消散案例
- 未完成案例
- 反例案例
- 无法判定案例
- 误用案例

每个案例至少记录：

```text
case_id:
材料边界:
使用场景:
七闸评分:
证据台账摘要:
机制候选:
判断档位:
行动上限:
实际后续:
误判点:
应降级处:
校准修订建议:
```

验证报告不得写“通过率证明框架可靠”。只能写：

- 哪些闸口最常失败。
- 哪些构念最难评分。
- 哪些锚点导致分歧。
- 哪些案例迫使判断降级。
- 哪些误用需要改模板或加红线。

## 评分者一致性

评分者一致性用于验证协议稳定性。

流程：

1. 建立训练集、校准集、正式评分集。
2. 至少两名评分者独立评分。
3. 记录原始分歧，不得只保留仲裁后结果。
4. 统计一致性时必须说明对象、维度和样本范围。
5. 一致性不足时，优先修订锚点，不得要求评分者迎合结论。

一致性结果不能证明现实判断为真。它只能说明：

- 规则是否足够清楚。
- 构念是否过度抽象。
- 锚点是否可操作。
- 哪些维度需要拆分、合并或退出。

## 反误用治理

v6.0 必须新增量化误用红线：

- 不得把半量化表包装成完整诊断。
- 不得把评分者一致性包装成现实正确性。
- 不得把案例库覆盖率包装成外部有效性。
- 不得把内部压力测试分数用于营销、认证或安全证明。
- 不得让付费方、委托方或被评估方单方控制证据入口、反馈入口和发布口径。
- 不得把低权力主体无法提交反例解释为“无反例”。
- 不得将 AI 自动评分用于强判断、公开定性或组织处置。

任何工具化版本必须显示：

```text
本工具只帮助整理结构判断，不替代现实调查、专业判断、受影响位置反馈、外部复核和最终责任承担。分数只能触发降级、复核或补证，不能单独授权处置。
```

## 工具原型范围

第一版可复用工具原型应包括：

1. 七闸半量化表：Markdown 模板和 JSON schema。
2. 证据台账扩展 schema。
3. 机制候选更新表。
4. 校准锚点卡模板。
5. 反例登记表。
6. 简单校验器：检查是否存在总分、缺撤回条件、缺行动上限、强判断缺来源台账、低权力入口缺失等问题。

第一版不做：

- 自动预测模型。
- 自动综合评分。
- 案例库成功率排名。
- 面向外部用户的公开评分系统。
- 与组织处置系统、绩效系统、审核系统或平台风控系统集成。

## 分阶段计划

### 阶段 1：设计冻结

产物：

- 本设计文档。
- v6.0 量化原则与边界章草案。
- 七闸半量化章草案。

完成标准：

- 明确可量化对象和不可量化对象。
- 明确数字只能承担的责任。
- 明确七闸、证据台账、机制候选、判断档位和行动上限的连接方式。

### 阶段 2：schema 与模板

产物：

- `seven-gates-quant.schema.json`
- `evidence-ledger-v6.schema.json`
- `calibration-anchor.schema.json`
- `mechanism-update.schema.json`
- `counterexample-register.schema.json`
- 对应 Markdown 模板。

完成标准：

- 每个 schema 有 fixtures。
- 校验器能阻断缺撤回条件、缺行动上限、强判断缺来源台账和总分输出。

### 阶段 3：案例库验证

产物：

- 3-5 个旧案例的半量化试跑。
- 分歧记录。
- 锚点修订建议。
- 误用风险清单。

完成标准：

- 记录至少一个不舒服案例或反例案例。
- 至少一次因反例导致降级、修订锚点或修改模板。

### 阶段 4：v6.0 文档成稿

产物：

- 可发布的《跨尺度结构诊断框架 v6.0》文档。
- v5.0 到 v6.0 的变更说明。
- v6.0 使用边界和工具化红线。

完成标准：

- v6.0 文档不把半量化层写成总分系统。
- 每个新概念或新工具都说明证据要求、误用风险和撤回条件。

### 阶段 5：可复用工具原型

产物：

- 模板、schema、校验脚本。
- README 使用说明。
- 小型 smoke test。

完成标准：

- 能对样例台账做结构校验。
- 能发现总分、缺撤回条件、缺行动上限和强判断越界。
- 明确声明工具不能替代现实复核。

## 开放问题

1. v6.0 是否采用 0-4 整数锚点，还是同时允许 fuzzy-set 0/0.25/0.5/0.75/1 隶属度？
2. 七闸半量化是否只用于后台，还是允许在审计版输出中显示？
3. 第一批案例库验证应选择关系、组织、公共制度各一个，还是先集中在组织修复场景？
4. v6.0 文档是否直接从 v5.0 DOCX 派生，还是先在 skill repo 中形成模块化 Markdown，再回写 DOCX？

## 初始推荐

采用 0-4 anchored rubric 作为第一版用户可读结构；内部可预留 fuzzy-set 隶属度字段，但不在前台展示。第一批案例验证选组织修复、亲密关系、公共制度各一个，以测试尺度迁移和高责任边界。v6.0 文档先在 skill repo 中模块化成 Markdown，稳定后再生成或回写 DOCX。
