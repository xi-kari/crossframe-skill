# Inquiry Protocol / 结构追问协议

## 1. 输入整理

读取上游材料后，先建立追问对象池：

| object_id | 类型 | 来源 | 当前状态 | 是否适合追问 |
| --- | --- | --- | --- | --- |
| CL | claim | claim ledger | pass / weak / partial / blocked | 是 / 否 |
| M | mechanism | mechanism_candidates | supported / competing / weak | 是 / 否 |
| C | concept | concept_contracts | pass / partial / fail | 是 / 否 |
| S | source | source ledger | complete / source-family-only / missing | 是 / 否 |
| R | review warning | review report | warning / failure / boundary | 是 / 否 |

### 完成态后续输入接管

若上一轮 suite 输出已经完成 `crossframe -> ... -> crossframe-essay -> crossframe-review`，并写入 `post_completion_inquiry_armed=true`，则完整链路完成后的任何实质后续用户输入都先进入本协议。不得要求用户再次说“继续追问”，也不得默认重新进入 essay。纯致谢、确认收到、结束语或无内容回应（如“谢谢”“好的”“明白了”“先这样”）不自动展开 inquiry；只轻量收束或结束本轮。若用户随后再次提出与上一轮相关的问题，再重新触发 inquiry。

先做上一轮上下文回收：

| context_item | 来源 | 用途 |
| --- | --- | --- |
| claim ledger | 上一轮命题台账 | 约束所有追问对象和判断强度 |
| mechanism_candidates | 上一轮机制候选 | 判断用户输入是在反证、迁移还是补证 |
| concept_contracts | 上一轮概念契约 | 防止概念滑义 |
| source ledger / v5-read-state-capsule | 上一轮来源与读态 | 识别证据缺口和降档条件 |
| insight dossier / article body | 上一轮底稿和正文 | 定位用户正在回应的表达段落 |
| review warning | 上一轮质量闸 | 把未闭合问题转成追问对象 |

如果用户后续只说“嗯”“然后呢”“那我呢”“继续”“所以怎么办”，也要把它解释为对上一轮输出的追问入口，并生成 3-5 个结构问题。只有用户明确说“新任务 / 换主题 / 退出追问 / 不接着上文”时，才解除完成态接管。

## 2. Sibling skill 知识库检索

如果上一轮上下文不足以生成高质量追问，或用户后续输入明显指向组织、公共、历史、读书、辩论、教学、案例、批判或短答复边界，可以做 `sibling_knowledge_retrieval`。

检索步骤：

1. 从用户当前输入和上一轮上下文中抽出检索目标，例如“组织迁移条件”“公共事实边界”“历史因果降级”“最强反方”“概念误读”。
2. 选择 1-3 个 sibling skill。不得调用全部 sibling skill。
3. 只读必要协议、references、templates；默认不读 examples 和 evals。
4. 生成 `retrieval_log`，记录检索了什么、为什么检索、能帮助生成哪些追问、不能证明什么。
5. 把检索结果转化为问题对象和风险边界，不转化为新结论。

常见路由：

| 用户后续输入 | 可检索 sibling skill | 用途 |
| --- | --- | --- |
| “那我们公司呢？” | `crossframe-org` | 组织迁移条件、责任链、授权链、反馈写回 |
| “这能公开说吗？” | `crossframe-public` / `crossframe-review` | 发布边界、风险语言、证据阶梯 |
| “历史上有没有类似的？” | `crossframe-history` | 史料档位、历史因果降级、类比边界 |
| “怎么反驳？” | `crossframe-debate` | 最强反方、隐藏前提、反证问题 |
| “这个概念我没懂” | `crossframe-teach` / `crossframe` | 概念误读、微练习、概念保真 |

如果检索后发现需要专项判断，回到 suite，让对应 sibling skill 正式处理。换言之：如果需要专项判断，回到 suite。inquiry 只能输出追问、边界和 `inquiry_delta`。

## 3. 追问模式

每轮只能选择 1-2 个主模式。

### A. 理解型追问

目标：帮助用户复述对象、机制和边界。

问题形态：这篇分析真正的中心判断是哪一条？它解释的是现象、机制、责任，还是行动边界？

### B. 反证型追问

目标：寻找最强反例和替代机制。

问题形态：如果要推翻 CL2，最有力的反证会来自哪里？M1 不能解释哪些现象？是否存在一个更低成本的解释？

### C. 补证型追问

目标：把“我觉得有道理”转成“还需要什么证据”。

问题形态：这条 claim 要升级，需要哪类 source_id？现在的来源只能支持背景，还是能支持机制判断？

### D. 迁移型追问

目标：把结构机制迁移到新对象，同时保留差异。

问题形态：这个机制迁移到另一个案例时，哪些条件必须相同？哪些条件一变，原判断就失效？

### E. 自我定位型追问

目标：帮助用户识别自己站在哪个位置，而不是直接给建议。

问题形态：你在这个结构里更像承接成本的人、定义规则的人、执行规则的人，还是旁观者？

### F. 行动边界型追问

目标：把行动冲动降为低条件行动或补证路径。

问题形态：这条判断最多允许什么行动？哪个事实没有确认之前不能行动？什么信号出现时必须停止？

### G. 概念保真型追问

目标：检查概念是否真的能承担判断。

问题形态：你现在用的是概念原义，还是表达转译？这个概念落回了哪些现实行为？它不能证明什么？

## 4. 问题数量

- 默认：3 个主问题 + 2 个可选深挖。
- 高压、脆弱、亲密关系、公共风险场景：最多 3 个问题。
- 研究、历史、组织、读书场景：可给 5 个问题。

不得一次性输出十几个问题，避免追问轰炸。

## 5. 追问格式

```markdown
## 结构追问

### Q1. [问题标题]

- 追问对象：CL / M / C / R
- 为什么问：
- 问题：
- 回答时可以从：
  - A
  - B
  - C
```

## 6. 用户回答后的处理

用户回答后，先摘要，不急着下结论。

```markdown
## 你的回答摘要

- 新增事实：
- 新增判断：
- 情绪/价值信号：
- 证据缺口：
- 可能改变的 claim：
- 不应升级的地方：
```

新材料必须标记为 `inquiry_delta`；若会改变原判断，生成 `inquiry_claim_delta` 并回到 claim ledger / review。

## 7. 禁止事项

- 禁止用追问施压用户接受结论。
- 禁止把追问伪装成引导式答案。
- 禁止在没有新证据时升级 claim。
- 禁止在用户脆弱、关系受伤或高责任场景中连续追问责任。
- 禁止把“你为什么不……”作为默认问法。
- 禁止把 sibling 知识库检索当成专项判断授权。
